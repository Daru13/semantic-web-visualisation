Visualising SPARQL responses

« Readable URIs »
We built a library to enable developers working with semantic web data to better visualise the structure and the statistical properties of SPARQL query responses (e.g. tables containing lots of URLs). Our contribution is twofold:
  • we enhanced the tabular representation of the raw response of the SPARQL query;
  • we built two visualisations to illusyrate the structure and the theme of sets of URIs.

Enhancing the raw response

We started by improving the representation of URIs in tje data table. We chose to hide the uninteresting, repetitive parts (such as the protocol) and to highlight the most meaningful ones for our target audience:
  • the domain, i.e. the name of the website, in bold;
  • the country as a flag, when available. We attempt to extract it from the URI by using different heuristics based on common URI patterns;
  • the path, which appears to usually describe what is the pointed resource. We highlight the structure the path by colouring slashes (which separate the different parts of the path) and almost hiding underscores (which usually act as whitespace).
We then added dashboards at the top of each column. They display basics statistics about the content of the column and allow to open more advanced visualisations, described below.

Visusalizing the structure of URIs

In order to better understand the structure of URI we built two visualisations.

The first one is a word cloud. It is displaying the keywords of a column organized from the most present to the least. A keywords is either the last part of the path of an URI, i.e. the name of the webpage, or keywords extracted from pure text by regular expressions. This visualisations allows to understand the dominant theme in a column. For example, in demo 1, for the genre column, we can read the most present genres of music in the database.

The second one is a Sankey diagram. It has three main columns: the subdomain, the domain plus tld and the path columns. For each elements of those columns (for example wikipedia.org), the more URIs with that element, the thicker and the more saturated the visual representation is. For two elements from two of those columns (for example dbpedia.org and ressource), the more URIs with both elements, the thicker and the more saturated the link between those two elements is. If one can understand how the URIs from on website are structured, one can understand how the website is structured.
